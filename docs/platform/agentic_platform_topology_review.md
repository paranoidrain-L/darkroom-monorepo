# Agentic Platform SubAgent 拓扑评估

## 目标

这份文档对 [agentic_platform_topology.md](agentic_platform_topology.md) 提出的 `SubAgent` 协作模式做一版简要评估。

评价重点不是复述拓扑，而是回答下面几个问题：

- 这套协作模式是否合理
- 它的核心优点是什么
- 它的主要落地风险是什么
- 对小团队是否现实
- 下一步要补哪些内容，才能从“概念正确”走到“可执行平台设计”

## 总体结论

这套协作模式是合理的，而且属于平台级正确设计，不是玩具式多 Agent 分工图。

它最大的优点是：

- 不是把 `SubAgent` 当作一组角色名，而是当作一组控制回路上的功能节点
- 同时覆盖了执行、协调、控制、审计、外部感知和政策约束六类能力
- 具备从单任务系统向平台系统演进所需的闭环结构

但它目前仍偏“拓扑级设计”。

如果要真正落地为一个可运营的 `Agentic Platform`，还需要继续补齐：

- 接口契约
- 权限边界
- 升级与阻断规则
- 指标体系
- 逻辑角色和运行单元的映射关系

## 一、这套模式为什么合理

### 1. 用 VSM 做骨架是正确的

把 `SubAgent` 拆成 `S1 / S2 / S3 / S3* / S4 / S5` 六类功能，是比“先起岗位名再凑职责”更稳的做法。

它能天然避免以下问题：

- 所有角色都既执行、又协调、又验收
- 执行链和控制链混在一起
- 平台看起来能跑，但没有稳态

### 2. 独立的 S3* 审计通道非常关键

很多多 Agent 设计只有：

- `Planner`
- `Worker`
- `Reviewer`

这种结构能跑，但很容易长期积累“伪成功”。

这份设计把 `S3*` 作为独立事实通道提出，是整份拓扑最有价值的部分之一。

它回答的是：

- 输出是不是真的可信
- 失败路径有没有被遮蔽
- 平台指标是不是在自我美化

### 3. S3 和 S4 的分离是平台思维，而不是任务思维

`S3` 关注当前稳态：

- 吞吐
- 成本
- 并发
- 失败恢复

`S4` 关注外部变化：

- 模型能力变化
- 工具链变化
- 需求变化
- 成本结构变化

这两个层如果不分开，系统通常会长期只优化当前，而失去未来适应能力。

### 4. S2 协调层是多 Agent 系统的必要层

这套设计没有把多 Agent 简化成“多个 Worker 并行跑”，而是明确保留：

- `Planner / Orchestrator`
- `Context Manager`

这是对的。

真实系统里，多 Agent 最常见的问题不是“没人执行”，而是：

- 重复劳动
- 上下文漂移
- 覆盖彼此结果
- 子任务顺序紊乱

这些都属于 `S2` 的责任。

## 二、这套模式的核心优点

这套协作模式的核心优点，可以概括为四点。

### 1. 闭环完整

它至少包含三条关键回路：

- 执行回路
- 控制回路
- 审计回路

这意味着平台不只是能产出结果，还能：

- 维持当前稳态
- 独立纠偏
- 吸收外部变化

### 2. 角色不是人设，而是功能节点

这使设计更适合平台，而不只是适合“给 Agent 起名字”。

它真正回答的是：

- 谁做事
- 谁协调
- 谁控制资源
- 谁独立验收
- 谁看外部世界
- 谁定义边界

### 3. 小团队也有最小可行形态

文档没有要求一开始就跑十几个独立 Agent，而是给出了最小可行能力集合：

- `Worker`
- `Coordinator`
- `Supervisor`
- `Tester / Auditor`
- `Researcher / Policy`

这个取舍是现实的。

### 4. 明确了哪些职责不应混在一起

尤其是下面几组边界，判断是正确的：

- `Worker` 和 `Tester`
- `Supervisor` 和 `Auditor`
- `Runtime` 和 `Policy`
- `Current Optimization(S3)` 和 `Future Adaptation(S4)`

这些边界一旦混掉，平台会很快出现目标扭曲。

## 三、主要落地风险

虽然拓扑本身合理，但要落地时有几个高风险点。

### 1. S2 和 S3 容易在实现里混掉

如果：

- `Planner / Orchestrator` 可以改任务优先级
- `Supervisor / Runtime Manager` 也可以改任务优先级

那最后它们就会冲突。

因此必须明确：

- `S2` 负责协作可运行
- `S3` 负责资源和稳态控制

两者都能“调整流程”，但调整权限不能重叠到没有边界。

### 2. S3* 容易沦为形式化审查

如果 `Tester / Auditor` 只能看执行者提交的摘要，而不能直接读取：

- logs
- traces
- artifacts

那它就不是独立审计，只是另一层汇报。

真正的 `S3*` 必须能绕过正常汇报链获得事实信号。

### 3. S4 很容易被边缘化

在资源紧张时，团队最容易牺牲的就是：

- `Researcher`
- `Scout`
- `Evaluator Designer`

短期看这会提升执行效率，长期看会让平台越来越脱离外部环境。

### 4. S5 如果只停留在文档层，会失效

`Policy / Goal` 如果只是写在文档里，而不落实成：

- 审批门槛
- 权限控制
- 风险阻断规则
- 资源预算

那么它不会真正改变系统行为。

### 5. 逻辑角色不等于运行单元

这份设计是合理的逻辑角色图，但不能直接等价为：

- 每个角色一个独立常驻进程
- 每个层都实时在线

对小团队而言，若一开始全做成独立运行单元，运维成本会迅速上升。

## 四、对小团队是否现实

结论是：现实，但前提是按“逻辑能力”先落地，而不是按“独立实例数”先落地。

更现实的第一版方式是只保留五个逻辑能力：

- `Worker`
- `Coordinator`
- `Supervisor`
- `Tester / Auditor`
- `Researcher / Policy`

也就是说：

- 逻辑上保留 `S1 / S2 / S3 / S3* / S4+S5`
- 运行上不必一开始就拆成八个长期常驻独立 Agent

因此，这套模式适合作为平台架构骨架，但小团队应避免一开始就过度实例化。

## 五、建议补充的四类设计内容

要把当前拓扑从“概念级正确”推进到“工程上可执行”，建议补四类东西。

### 1. 接口契约

要明确每层：

- 输入是什么
- 输出是什么
- 可以改什么
- 不可以改什么

尤其应写清楚：

- `S2` 可以拆任务，但不能改全局政策
- `S3` 可以降级、限流、回退，但不能伪造通过
- `S3*` 可以否决结果，但不直接改写执行产物

### 2. 升级与阻断规则

至少要定义：

- 什么情况下 `Worker` 直接失败
- 什么情况下 `Orchestrator` 重试
- 什么情况下 `Supervisor` 降级
- 什么情况下 `Auditor` 升级到 `Policy`

没有这些规则，系统会在失败时进入模糊状态。

### 3. 指标体系

不能只看：

- 成功率
- 吞吐
- 平均时延

还要看：

- 审计打回率
- 返工率
- 静默失败率
- 伪成功率
- 外部变化适配时延

这些指标决定 `S3*` 和 `S4` 是否真正有抓手。

### 4. 逻辑角色到运行单元的映射

需要进一步回答：

- 哪些是独立 Agent
- 哪些只是同一 Agent 的不同 mode
- 哪些是离线流程，而不是实时 Agent
- 哪些能力应该嵌入共享子系统，而不是独立对话体

## 六、推荐的落地判断

如果把这份拓扑作为 `Agentic Platform` 的协作模式设计，我的判断是：

- 架构方向正确
- 闭环结构完整
- 足够适合作为平台设计的上层 operating model

但它还不是最终可执行规格。

它更像：

- 一份高质量的系统拓扑与功能分层设计

而不是：

- 一份已经可直接实现的接口与运行时规范

## 七、一句话结论

这套协作模式是平台级正确的，它最大的优点是闭环完整，最大的落地风险是层间边界不够硬。

如果下一步继续补上：

- 权限边界
- 输入输出契约
- 升级规则
- 指标体系

它就能从一份好看的拓扑图，演进成一套真正可运行的 `Agentic Platform` 协作模型。
