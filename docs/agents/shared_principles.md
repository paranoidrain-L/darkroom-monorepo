# Agent 协作共享原则

本文档是 Planner、Worker、Tester 三个角色共享的基础原则。各角色方法论引用本文档，而非重复。

## 一、失败路径设计原则 {#失败路径设计原则}

失败路径必须和主路径同等设计。

必须主动设计的失败模式：

- AI backend 初始化失败
- 返回非法输出
- 检索不到证据
- 样本不足
- 路径配置错误
- webhook 失败
- sender 直接抛异常
- 历史数据部分损坏

常见降级模式：

- 返回 fallback，而不是抛异常
- 拒答，而不是胡答
- 事实汇总，而不是伪洞察
- 标记为 pending / failed，而不是中断主流程

判断标准四问：

1. 失败是否可观测
2. 失败是否可重试
3. 失败是否会污染已有数据
4. 失败是否会打断不该被打断的主流程

## 二、静默错误与伪成功 {#静默错误与伪成功}

静默错误：不报错但污染系统语义的问题。

典型案例：

- `all_articles` 和 `report_articles` 混淆
- 缺失 DB 路径被静默创建为空库
- citation URL 不来自真实命中文章
- sender 异常导致主流程中断

为什么比显式异常更危险：

- 用户会相信错误输出
- 测试可能仍然大面积通过
- 后续 phase 会建立在错误资产之上

## 三、测试四分类结构 {#测试四分类结构}

测试不只看数量，要看结构是否覆盖风险。

### 1. Contract Tests

验证输入输出 schema、字段语义、状态迁移是否稳定。

### 2. Compatibility / Migration Tests

验证旧数据能否读取、迁移、回填，且语义不变。

### 3. Failure / Fallback Tests

验证初始���失败、非法输出、空输出、异常、限流、缺库等场景不会失控。

### 4. Golden / Evaluation Tests

验证 search、RAG、insights 这类逻辑在固定语料上的行为是否稳定。

测试优先级：

1. 会破坏语义的路径
2. 会中断主流程的路径
3. 兼容和迁移路径
4. 主路径 happy case

## 四、Findings 严重级别排序 {#findings-严重级别排序}

Findings 按阻断性排序：

1. 会中断主流程的问题
2. 会污染数据语义的问题
3. 会破坏兼容 / 迁移的问题
4. 会误导用户但主流程仍能跑的问题
5. 文档与边界不一致的问题
6. 测试缺口但暂未形成已复现 bug 的问题

阻断判断三问：

- 它会不会让 phase 的最小闭环失效
- 它会不会让失败路径不可控
- 它会不会让用户看到"伪成功"

只要有一个答案是"会"，通常就应阻断。

## 五、文档一致性要求 {#文档一致性要求}

至少同步三类文档：

- README
- phase 设计文档
- roadmap / 边界说明

文档必须明确：

- 已落地能力
- 当前最小边界
- 明确未做项
- 测试门禁

禁止：

- 把"最小可用版"写成"完整能力"
- 把"后续增强项"写成"当前已具备"

## 六、通用检查清单 {#通用检查清单}

每次 phase 交付或验收前，至少过一遍：

### 闭环

- 最小能力链路是否真的打通
- 有没有依赖人工脑补的隐含步骤

### 失败路径

- 初始化失败怎么办
- 下游抛异常怎么办
- 空结果怎么办
- 非法输入怎么办

### 数据语义

- 新旧数据是否混淆
- 关键字段语义是否稳定
- 状态迁移是否可解释

### 兼容迁移

- schema version 是否更新
- migration 是否覆盖老版本
- 是否需要 backfill

### 测试

- 定向测试是否覆盖新缺口
- 全量回归是否通过

### 文档

- README 是否同步
- phase 文档是否同步
- 未做边界是否写清楚
