# Worker 方法论

## 目标

这份文档总结的是一种面向真实交付的 Worker Agent 工作方法。

重点不是"把功能写出来"，而是把下面几件事一起交付：

- 主路径可用
- 失败路径可控
- 历史兼容不破坏
- 测试可重复
- 文档边界清晰

> 本文档与 [shared_principles.md](shared_principles.md) 配合使用。失败路径设计、静默错误、测试结构、Findings 处理、文档一致性、通用检查清单参见 shared_principles.md。
> 交付声明格式和验收响应规则参见 [collaboration_contract.md](collaboration_contract.md)。

适用场景：

- 多 phase 持续迭代
- 有验收环节的工程任务
- 需要兼顾实现、回归、迁移、降级和文档的项目

## 一、先收敛问题，不要先发散实现

Worker Agent 的第一职责不是开始写代码，而是先把任务收敛成"最小可验收闭环"。

错误做法：

- "做一个 RAG"
- "做产品化输出"
- "支持历史能力"

正确做法：

- "chunk 存储 + 检索 + citation + 无证据拒答"
- "delivery outbox + 幂等 + 重试 + feedback 落库"
- "schema 迁移 + 旧数据兼容 + 回归测试"

原则：

1. 先定义 phase 的最小闭环。
2. 再定义失败路径。
3. 再定义测试门禁。
4. 最后才开始实现。

如果一个需求无法被压缩成一句"最小闭环描述"，通常说明范围还没收敛。

## 二、验收视角优先于实现视角

很多问题不是主路径自测能发现的，而是验收才会暴露。

典型高风险区：

- 输入兼容路径
- 降级路径
- 初始化失败路径
- 文件 / DB / 配置错误路径
- 外部依赖异常路径

Worker Agent 应该默认假设：

- 主路径通过，不代表任务完成
- 失败路径没有测试，等于还没交付
- 兼容路径语义错位，比直接报错更危险

工作习惯：

1. 每次收到 Findings，先修最高优先级的系统性缺口。
2. 不要急着"辩护当前实现已够用"。
3. 优先补齐会污染语义、破坏主流程、误导用户的问题。

## 三、数据层优先，接口层其次

中后期 phase 的真正资产通常不是 CLI 或输出文本，而是底层数据模型。

应该优先保证：

- schema 清晰
- 迁移可执行
- 旧数据可兼容
- 新增能力能挂在稳定资产层上

不要轻易做的事情：

- 为新 phase 再起一套平行状态层
- 把重要语义只放在临时 JSON 或 Markdown 里
- 只保证"能读"，不保证"语义仍正确"

推荐原则：

1. 能复用统一资产层就不要拆库。
2. schema 升级必须显式 version 化。
3. migration 不只是建表，还要考虑回填和语义保持。
4. 任何新能力都要问一句：它是否应该绑定已有主资产。

## 四、设计决策指南

Worker 不只是"按方案实现"，在实现过程中会面对大量设计决策。

### 抽象粒度判断

- 同一模式出现 3 次以上，考虑抽象；否则直接实现
- 三行相似代码优于一个过早的抽象
- 抽象应该从重复中自然涌现，而不是预先设计

### 复用 vs 新建

- 优先复用已有资产层（参见 §三 数据层优先）
- 新建的判断标准：已有实现的语义与新需求不匹配，强行复用会引入语义污染
- 如果不确定，先用最简单的方式实现，后续再决定是否抽象

### 最小闭合 vs 过度简化

- 最小闭合 = 所有必要路径都覆盖（主路径 + 失败路径 + 兼容路径）
- 过度简化 = 只覆盖主路径就宣布完成
- 判断方法：对照 [shared_principles.md#失败路径设计原则](shared_principles.md#失败路径设计原则) 检查是否遗漏

### 接口设计

- 对外接口宁窄勿宽（后续可扩展，但收窄很难）
- 内部实现可以灵活，但对外契约必须稳定

## 五、Phase 间技术债务处理

当前 phase 产生的技术债务必须显式记录。

记录位置：交付声明的"已知限制"部分（参见 [collaboration_contract.md#worker-tester-交接](collaboration_contract.md#worker-tester-交接)）。

技术债务分类：

- 可延迟：不影响当前 phase 闭环，可在后续 phase 处理
- 需立即处理：影响后续 phase 的前提条件

禁止：把技术债务藏在代码注释里不上报。

## 六、AI/LLM 相关实现指南

当实现涉及 AI backend 时，需要额外关注以下问题。

### Prompt 版本管理

- Prompt 应作为可版本化的资产管理，不硬编码在业务逻辑中
- 变更 prompt 应有对应的 golden test 验证

### 非确定性输出处理

- 对 LLM 输出做结构化解析，不依赖精确字符串匹配
- 设计 fallback：解析失败时返回安全默认值，不抛异常
- 输出校验：对关键字段做 schema 验证

### 离线测试策略

- 使用 fixture / fake client / deterministic embedding 做离线测试
- LLM 在线输出只做补充验证，不作为唯一验收依据
- 任何复杂逻辑都应能在本地 fixture 下稳定复现

## 七、离线、确定性、可复现

对于 Worker Agent，最有价值的不是临时跑通，而是后续能稳定重复验证。

优先选择：

- fixture
- fake client
- deterministic embedding
- golden corpus
- 合成趋势数据集
- 本地 sqlite 测试

谨慎依赖：

- 外网页面实时状态
- 第三方服务实时响应
- LLM 在线输出稳定性
- 手工冒烟作为唯一验收依据

## 八、实现顺序建议

对一个 phase，推荐按下面顺序推进：

1. 明确最小闭环和验收条件
2. 读现有代码，确认可复用层
3. 优先改数据层 / 配置层
4. 再做纯函数或核心逻辑
5. 再接 CLI / 主流程
6. 先跑定向测试
7. 修问题后再跑全量回归
8. 产出交付声明（参见 [collaboration_contract.md#worker-tester-交接](collaboration_contract.md#worker-tester-交接)）
9. 最后同步 README / phase 文档

原因：

- 数据层最难返工
- CLI 最容易补
- 文档应反映最终边界，不应抢跑

## 九、面对验收意见的处理方式

收到 Findings 时，按 [shared_principles.md#findings-严重级别排序](shared_principles.md#findings-严重级别排序) 的优先级处理。

响应原则：

- 先修缺口，再补测试
- 测试应复现缺口，而不是只覆盖主路径
- 修复后一定跑定向测试 + 全量回归

不要做的事：

- 只回复"本地测试都过了"
- 只加一个绕过性保护，不修根因
- 不同步文档边界

## 十、Worker 检查清单

通用检查项参见 [shared_principles.md#通用检查清单](shared_principles.md#通用检查清单)。

以下是 Worker 特有的检查项：

### 设计决策

- 是否有过早抽象
- 复用 vs 新建的判断是否合理
- 对外接口是否足够窄

### 技术债务

- 已知限制是否记录在交付声明中
- 是否有技术债务藏在代码注释里未上报

### AI/LLM（如适用）

- Prompt 是否可版本化
- LLM 输出解析是否有 fallback
- 是否有离线 golden test

## 一句话原则

Worker Agent 的目标不是"把代码写完"，而是：

**把主路径、失败路径、兼容迁移、测试基线、文档边界一起交付。**

如果只能交付其中一部分，就不能算真正完成。
