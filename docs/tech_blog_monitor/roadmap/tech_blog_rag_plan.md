# Tech Blog Historical Archive and RAG Plan

## 背景

`tech_blog_monitor` 当前已经具备以下基线能力：

- 多 RSS 源并发抓取
- Markdown 报告输出
- JSON 结构化输出
- 增量状态记录与新增文章视图
- 历史归档（按天 / 周）
- Phase 1 sqlite 历史资产存储
- AI 摘要与趋势分析降级处理

如果只关注“当天 / 本周新文章”，产品价值主要体现在信息聚合。
如果进一步沉淀历史文章，则可以逐步演进为团队内部的技术情报库和研究辅助系统。

因此，历史文章不应只作为归档文件存在，而应作为可检索、可追溯、可分析的数据资产长期维护。

## 目标

将 `tech_blog_monitor` 从“RSS 报告生成器”逐步升级为：

- 历史技术文章归档库
- 可检索的专题知识库
- 面向技术趋势分析的 RAG 系统
- 团队内部的研究与选题辅助工具

## 历史文章的产品价值

### 1. 主题沉淀

持续积累某个方向的文章，例如：

- AI Agent 工程实践
- 推理时扩展
- 多模态模型
- 自动驾驶 / 感知
- AI 基础设施

沉淀后的价值不在于单篇文章，而在于形成长期专题知识库。

### 2. 趋势追踪

支持回答：

- 某个技术方向过去 3 个月如何演进
- 某个公司最近持续关注哪些主题
- 某个主题是短期热点还是持续趋势

### 3. 对比分析

支持回答：

- OpenAI、Meta、DeepMind 对同一主题的关注点差异
- 学术源和工业源在某个方向上的信号是否一致

### 4. 研究辅助

支持内部用户快速找到：

- 某个问题相关的历史文章
- 值得精读的核心资料
- 某个主题的代表性观点和原文出处

### 5. 选题与分享支持

支持自动生成：

- 本周最值得分享的主题
- 某个方向最近一个月的重要进展
- 某个专题的阅读清单

## 是否适合做成 RAG

结论：适合，而且有明显价值。

但不建议直接跳到“全量文章一股脑做向量检索 + 问答生成”。

更合理的路线是分层建设：

1. 历史文章归档
2. 可检索知识库
3. 混合检索
4. 问答式 RAG
5. 趋势分析与主题图谱

## 建议的数据模型

每篇文章建议至少保存以下字段：

- `id`
- `title`
- `url`
- `source_name`
- `category`
- `published_at`
- `fetched_at`
- `rss_summary`
- `clean_text`
- `ai_summary`
- `tags`
- `language`
- `content_hash`

如果要做 RAG，还需要额外存储：

- `chunks`
- `chunk_index`
- `embedding`
- `topic`
- `time_bucket`

## 历史文章系统的分阶段路线

### Phase A：历史归档库

目标：先把文章变成长期资产，而不是一次性报告输入。

交付：

- 原文或清洗后的全文落库
- 文章元数据归档
- 文章去重
- 历史记录持久化
- 支持按时间 / 来源 / 分类筛选

建议能力：

- 支持按天 / 周 / 月归档
- 支持全文检索
- 支持查看单篇文章详情

阶段价值：

- 为后续检索和分析提供基础数据
- 解决“报告生成完就丢失上下文”的问题

### Phase B：可检索知识库

目标：让历史文章可被高效查询和复用。

交付：

- 支持按关键词搜索
- 支持按来源过滤
- 支持按时间范围过滤
- 支持按主题 / 分类聚合

建议能力：

- “过去 30 天关于 Agent 的文章”
- “OpenAI 最近关于企业 AI 的文章”
- “自动驾驶 / 3D 感知相关文章列表”

阶段价值：

- 先建立稳定检索体验
- 不急于引入 LLM 生成，先保证结果准确

### Phase C：混合检索

目标：在关键词检索之外，引入语义召回能力。

交付：

- 文章 chunk 切分
- embedding 建库
- 向量检索
- 关键词检索 + 元数据过滤 + 向量召回的混合检索

推荐检索条件：

- 关键词过滤
- 时间过滤
- 来源过滤
- 分类过滤
- 向量相似度召回

阶段价值：

- 提升模糊问题的召回效果
- 适合“我不知道精确关键词，但知道主题”的场景

### Phase D：问答式 RAG

目标：从“检索文章”升级到“基于文章回答问题”。

交付：

- 问题理解
- 检索命中文章 / chunk
- 基于命中文章生成回答
- 回答附证据列表和原文链接

建议问答场景：

- 最近半年有哪些文章讨论 inference-time scaling
- Meta 和 LangChain 对 enterprise agents 的关注重点有什么差异
- 自动驾驶感知团队最近值得关注的多模态评测文章有哪些

阶段价值：

- 让系统从资料库升级为研究助手

### Phase E：分析型产品

目标：从“回答问题”进一步升级为“生成洞察”。

交付：

- 主题聚类
- 趋势变化分析
- 时间线生成
- 多来源对比
- 热点信号提取

建议分析输出：

- 某主题过去 90 天演进图
- 某公司持续关注方向
- 本周 / 本月技术热点变化
- 同一主题的行业共识与分歧

阶段价值：

- 从检索工具升级为技术情报系统

## RAG 设计建议

### 1. 不要只存摘要

摘要适合阅读，不适合高质量检索。

建议同时保存：

- RSS 摘要
- 清洗后的正文
- AI 摘要

### 2. 必须重视时间维度

技术文章时效性强，RAG 不能只看语义相似度。

至少应支持：

- 时间过滤
- 时间衰减排序
- 最近优先

### 3. 必须做混合检索

仅靠向量召回通常不够稳定。

建议采用：

- 关键词检索
- 元数据过滤
- 向量检索
- 重排

### 4. 必须做去重

否则转载、镜像、重复信号会污染结果。

建议去重依据：

- URL
- 标题近似度
- 内容 hash

### 5. 回答必须附出处

RAG 回答必须带：

- 文章标题
- 来源
- 发布时间
- 原文链接

否则产品很难获得信任。

## 可能的使用场景

### 场景 1：技术趋势问答

示例：

- 最近 3 个月 Agent 工程有哪些关键变化
- 推理时扩展相关的文章最近有什么共识

### 场景 2：来源对比

示例：

- OpenAI、Meta、DeepMind 最近对企业 AI 的表述有什么差异

### 场景 3：团队定制推荐

示例：

- 自动驾驶感知团队最近最值得读的 5 篇文章
- 基础设施团队本周应关注哪些内容

### 场景 4：内部分享支持

示例：

- 生成某个专题的阅读清单
- 生成内部分享的背景材料和参考文章

## 风险与注意事项

### 1. 数据质量风险

- RSS 摘要不完整
- 正文抓取失败
- 时间字段不规范
- 同一文章多处重复

### 2. 检索质量风险

- 向量召回命中不准
- 热门关键词导致噪音过多
- 时间维度被忽略

### 3. 生成质量风险

- 回答泛化
- 证据不足
- 引用不准确

### 4. 成本风险

- 全量 embedding 成本
- 历史归档存储成本
- 增量更新和重建索引成本

## 推荐实施顺序

1. 先做历史归档库
2. 再做可检索知识库
3. 然后做混合检索
4. 最后做问答式 RAG
5. 成熟后再做趋势分析和主题图谱

## 推荐结论

`tech_blog_monitor` 非常适合发展成一个历史文章知识库，并进一步演进为技术情报型 RAG 系统。

但最重要的前提不是“尽快上向量库”，而是先把历史文章的数据结构、归档、去重、时间维度和检索基础打牢。

更务实的路线是：

- 先从历史归档和检索做起
- 再逐步引入 RAG
- 最终升级为可回答问题、可生成洞察的研究辅助产品
