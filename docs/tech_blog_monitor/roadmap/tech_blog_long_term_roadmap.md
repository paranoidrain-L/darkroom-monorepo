# Tech Blog Monitor Long-Term Roadmap

更新时间：`2026-04-18`

## 目标

这份文档用于整理 `tech_blog_monitor` 的长期规划。

它回答的不是“下一步修哪个模块”，而是下面四个更长期的问题：

- 这个项目长期要变成什么
- 它是不是一个 Agent 系统
- 如果是，它应该是什么类型的 Agent
- 它和 `VSM` 骨架之间的关系是什么

## 一句话结论

`tech_blog_monitor` 的长期方向，不是做成一个通用聊天 Agent，也不是继续停留在“RSS 抓取脚本”层面。

它更合理的长期定位是：

- 一个面向内部研发团队的技术情报 `Knowledge Agent`

它负责：

- 持续感知外部技术信息
- 把信息沉淀为可检索、可验证、可运营的知识资产
- 提供检索、问答、趋势洞察和运营闭环
- 在此基础上，逐步演进为单领域的 Agent 化知识系统

## 当前定位

截至 `2026-04-18`，当前系统更准确的定义是：

- 一个已经完成现代化底座升级的后端型技术内容管线
- 一个 `agent-ready` 的知识系统雏形
- 还不是一个完整闭环的 Agent 平台

当前已经具备的核心能力：

- 多源 RSS 抓取与增量更新
- 正文抽取与质量判断
- enrichment / search / QA / insights
- sqlite / PostgreSQL-ready 数据面
- 本地优先的 observability / task / orchestration 底座

当前还不应过度宣称的部分：

- 还没有完整的 retrieval evaluation 与高质量 ranking 主路径
- 还没有充分成熟的站点级正文规则库
- 还没有完善的 article-level 运营修复闭环
- 还没有前端产品层的稳定交互体验
- 还没有形成完整的 `S3* / S4 / S5` 闭环

## 北极星定义

长期看，这个项目要从“内容抓取系统”演进为“技术情报系统”，再进一步演进为“技术情报 Agent”。

北极星不是“抓到更多文章”，而是：

- 让内部团队能持续获得高质量、低噪音、可追溯的外部技术情报
- 让这些情报沉淀为长期知识资产，而不是一次性报告
- 让用户可以围绕这些资产进行检索、问答、订阅、反馈和任务化运营
- 让系统具备一定程度的自主感知、任务调度、质量校验与策略约束能力

## 长期演进路径

## 阶段 1：知识资产系统

这是当前到未来 `1-3` 个迭代周期内最重要的阶段。

目标是把系统从“能跑通”推进到“知识质量可控、运营可恢复、查询可用”。

核心工作：

- 完成 retrieval eval baseline，并以此驱动 ranking / embedding 升级
- 为高价值来源补站点级正文规则
- 把 delivery 做到最小异步化或可恢复化
- 补 article-level 修复任务，如 `reextract_content` / `reenrich_articles`
- 完成 `P3`，补齐前端入口或更稳定的产品化访问面

阶段产出应当是：

- 内容质量不再主要依赖人工抽查
- 问答结果可以通过基线评测持续改进
- 失败后的修复不必依赖整次 run 全量重做
- 内部用户能把它当成一个持续可用的技术知识底座

## 阶段 2：内部技术情报产品

这是中期阶段，重点不再是“补底座”，而是“形成稳定产品闭环”。

目标是把知识资产真正产品化。

核心工作：

- 构建更清晰的内部使用入口，包括 run、article、search、QA、insights、ops
- 补用户反馈、收藏、标签、订阅、来源质量管理等产品能力
- 建立来源组合策略，不只追求“更多 feed”，而是追求“更高价值 source portfolio”
- 建立质量看板，包括抓取成功率、正文命中率、QA 命中率、delivery 恢复率
- 建立更稳定的数据运营流程，如重建索引、批量回填、低质量内容巡检

阶段产出应当是：

- 系统不只是生成报告，而是提供长期可查询、可运营、可复盘的技术情报服务
- 用户开始围绕这个系统形成稳定使用习惯
- 运维和质量改进由“人工经验”逐步转为“指标驱动”

## 阶段 3：单领域 Knowledge Agent

当知识资产、检索质量、运营闭环都达到可用线之后，才适合进入更明确的 Agent 化阶段。

这里的 Agent，不是通用聊天机器人，而是：

- 一个单领域、面向技术情报任务的 `Knowledge Agent`

它应具备的能力：

- 主动追踪重点主题、来源和信号变化
- 将用户问题改写为检索任务、比较任务和归纳任务
- 根据上下文生成专题摘要、趋势说明和阅读建议
- 对低质量结果、缺失结果和冲突结果触发补抓、重排或人工复核建议
- 按策略自动执行部分低风险任务

这时系统的输出将不再只是：

- 一份日报或一个搜索结果

而会更像：

- 一名持续工作的技术情报助理

## 阶段 4：接入更大范围的 Agentic Platform

更长期看，`tech_blog_monitor` 可以成为更大平台中的一个领域 Agent。

它适合承担的角色不是总控调度者，而是：

- 外部技术信息感知节点
- 技术知识资产与检索节点
- 技术趋势和来源质量的评估节点

它可以向更大平台提供：

- 高质量知识上下文
- 技术趋势摘要
- 来源可信度与主题热度信号
- 可调用的检索、QA、重抽取、重分析等任务能力

## 6 个月路线

未来 `6` 个月更实际的目标不是“平台化”，而是把系统做成可靠的内部产品底座。

建议重点放在：

- retrieval evaluation、ranking、embedding 主路径升级
- 高价值源正文规则与质量 rebaseline
- article-level 修复任务与 delivery 恢复能力
- 前端或内部操作面补齐
- run / article / source / task / ops 的稳定 API 面
- PostgreSQL 主路径进一步实化，减少兼容层长期负担

`6` 个月后的合理状态应是：

- 查询质量明显优于当前 baseline
- 正文抽取对核心来源更加稳定
- 失败恢复和运营修复已形成日常工作流
- 系统对内部用户来说已经是“工具”，而不是“脚本”

## 12 个月路线

未来 `12` 个月的目标应从“产品可用”推进到“Agent 可用”。

建议重点放在：

- 建立主题级长期记忆，而不只是 article 级资产
- 建立事件、实体、主题之间的结构化关联
- 建立更强的 query planning / retrieval planning / answer grounding
- 建立持续评测体系，包括 QA、insights、正文质量、来源质量
- 建立策略层与风险边界，明确哪些任务允许自动执行，哪些需要审批
- 把系统作为一个领域 Agent 接入更大协作系统

`12` 个月后的合理状态应是：

- 用户可以把它视为“技术情报助手”
- 系统具备一定程度的主动性和任务闭环能力
- 但它仍是单领域 Agent，而不是通用 Agent 平台

## Agent 定位

如果把它定义为一个 Agent，那么它的定位应当明确为：

- 内部技术情报 `Knowledge Agent`

它的边界也应当明确：

它应该做的事：

- 抓取、清洗、理解外部技术内容
- 建立技术知识资产
- 提供检索、问答、趋势总结和来源管理
- 基于策略触发低风险运营动作

它不应该直接扩展成的方向：

- 通用聊天机器人
- 通用代码 Agent
- 全能办公助手
- 一开始就承担跨域总控调度平台

## 用 VSM 看当前状态

如果用 `VSM` 骨架看当前系统，可以认为：

- `S1`：较强
- `S2`：中等
- `S3`：中等
- `S3*`：弱到中等
- `S4`：较弱
- `S5`：较弱

### S1：执行层

当前已经比较完整：

- 抓取
- 正文抽取
- enrichment
- search / QA / insights
- delivery / feedback

### S2：协调层

已经有初步雏形：

- CLI / API / task runner / local scheduler / orchestration adapter

但还缺：

- 更明确的任务编排边界
- 更清晰的上下文与任务交接格式
- 更成熟的 article-level 运维任务模型

### S3：控制层

当前已经开始建立：

- observability
- ops summary
- task records
- local scheduler

但还缺：

- 更强的资源控制、降级和恢复策略
- 更明确的运行预算和优先级策略
- 更平台化的 runtime control

### S3*：独立审计层

这是当前明显偏弱的一层。

虽然已经有：

- 测试
- 基本运维指标
- 一些质量判断

但还缺：

- retrieval / QA 的持续评测基线
- 正文质量与来源质量的系统性审计
- 独立于主流程汇报的质量反馈通道

### S4：外部感知层

这层目前只有很弱的雏形。

当前系统能抓“外部内容”，但还没有充分形成：

- 对模型能力变化的跟踪
- 对来源组合价值的持续优化
- 对用户需求变化的系统吸收
- 对评测集和基线的持续演进

这意味着它目前更像内容系统，而不是成熟的适应性 Agent。

### S5：政策与目标层

这层目前也较弱。

当前虽然有阶段规划，但还缺：

- 明确的长期目标函数
- 自动动作审批边界
- 高风险行为阻断规则
- 平台身份和边界的制度化表达

## 长期建设重点

如果只保留最关键的长期建设主线，应该是下面五条：

1. 把知识质量做实，而不是继续堆功能入口
2. 把评测、审计、反馈做成闭环，而不是靠主观判断
3. 把任务与运维能力做成系统级能力，而不是手工补救
4. 把产品入口做稳定，再谈更强 Agent 化
5. 把 `S3* / S4 / S5` 补齐，避免系统只会“跑流程”

## 明确不做的事

为了控制长期路线，不建议把项目带向下面这些方向：

- 过早追求完整通用 Agent 平台
- 在 retrieval 质量还不稳时先堆复杂前端
- 在任务和恢复链路还不稳时先上复杂编排系统
- 把所有新能力都做成一次性大改，而不是渐进式替换
- 为了“看起来像 Agent”而引入没有闭环支撑的自治行为

## 最终判断

长期来看，`tech_blog_monitor` 应该演进成：

- 一个面向内部研发团队的技术情报 `Knowledge Agent`

但从现在到那个状态之间，仍然要先走完下面这条路径：

`RSS/Blog Monitor -> Knowledge Asset System -> Internal Tech-Intelligence Product -> Single-Domain Knowledge Agent -> Agentic Platform 中的领域节点`

这条路径的关键，不是先堆更多 Worker，而是先把：

- 知识质量
- 检索质量
- 运营修复能力
- 观测与审计
- 策略边界

真正做成闭环。
